## Контекст бизнеса

АГР ПАРТС — оптовый поставщик запасных частей для сельхозтехники и промышленного оборудования.

**Марки:** John Deere, Claas, AGCO, Fendt, Valtra, Massey Ferguson, Siemens, ABB, Schneider Electric, Festo, Allen Bradley

**Клиенты:** агрохолдинги, фермерские хозяйства, торгующие организации (B2B)

**Типовой цикл заказа:**
1. Клиент присылает тендер или заявку → менеджер принимает в ERP
2. Закупщик формирует запрос поставщику (ЗПО) → объединяет в Excel-файл (REQ) → отправляет поставщику
3. Поставщик отвечает ценами и сроками → закупщик актуализирует, размещает заказ по проформе
4. Поставка: Transit → Контрольный лист (физическая приёмка) → Склад
5. Бэкордер: если позиция не пришла — ведётся в отдельном модуле

## 1. Запросы (Quotation)

**Бизнес-логика:**  
Закупщик получает от менеджеров ЗПО (заявки поставщику по одной позиции/группе позиций). Несколько ЗПО объединяются в Excel-файл REQ, который отправляется поставщику по email. После получения ответа поставщик актуализирует цены и статусы.

**Страницы:**

### `page-quotation` — список запросов

Три таба:
- **Активные** — ЗПО от менеджеров, ещё не отправленные поставщику. Статусы: Обычный (зелёная точка) / Срочный (красная точка)
- **В работе** — отправленные REQ-файлы, ведётся переписка. Колонки: № Запроса | Поставщик | Марка | ЗПО | Позиций | Сумма EUR | Статус
- **Обработанные** — закрытые запросы

### `page-quotation-detail` — детальная страница REQ

Пример: REQ 12346, поставщик Hans Mueller & Co., марка AGCO

**Шапка:**
- Строка 1: № Запроса | Марка | Сумма РРЦ | Позиций | Сумма EUR | Ожидаю | badge статус
- Строка 2: Клиент | Клиентский номер
- Строка 3: Инициатор | Тип заказа | СРОЧНО (если актуально)
- Файл запроса: chip с файлом REQ12346_Hans-Mueller_02.06.xlsx

**Таблица позиций:**  
Колонки: # | Артикул | Наименование | Кол-во | Amount EUR (цена 1 шт.) | Total EUR (Amount × Кол-во) | Статус  
Поиск по артикулу: `qdSearch()` фильтрует tbody по `data-art`

**Переписка с поставщиком:**  
Под таблицей — секция с цепочкой писем (исходящие/входящие). Expand/collapse по клику (`qdToggleMsg`). Форма compose: тема, поставщик, текст, прикрепить файл. Чипы вложений.

### `page-new-request` — форма нового запроса

Порядок полей: **Марка → Поставщик → Поставка**  
При выборе марки автоматически фильтруются поставщики (`nrFilterSuppliers`):
- John Deere → Midwest Parts GmbH, Hans Mueller & Co.
- AGCO → Hans Mueller & Co.
- Claas / Fendt → AgriTech Europa BV

Панель «Письмо поставщику»: тема письма, текст, прикрепить файл.  
Модал «Сформировать запрос (Excel)»: 2 опции — Скачать файл / Прикрепить к письму (показывается chip вложения).

## 2. Заказы (Orders)

**Бизнес-логика:**  
После согласования с поставщиком размещается заказ по проформе. Заказ проходит полный жизненный цикл от размещения до физической приёмки на складе EU и перевода в завершённые.

### Жизненный цикл заказа (6 этапов)

```
Подтвержден → Заказан → Прибыл → Комплектация → Таможня → Склад РФ
```

| Этап | Триггер | Тип |
|---|---|---|
| Подтвержден | Поставщик прислал подтверждение | Полуавтомат |
| Заказан | Оплата отправлена, товар пошёл в сборку | Полуавтомат |
| Прибыл | Товар прибыл на склад EU поставщика | Полуавтомат |
| Комплектация | Контрольный лист открыт и ведётся | Автомат (по КЛ) |
| Таможня | Контрольный лист завершён + утверждён | Автомат (по КЛ) |
| Склад РФ | ТСД-сканирование на российском складе | Автомат (по ТСД) |

### Три уровня сверки

**Сверка 1 — Предварительная («Обновить статусы»)**  
Производится вскоре после размещения заказа. Уточняем у поставщика статус позиций: что подтверждено, что переносится, что отменено. Реализована как попап с текстовым полем (WhatsApp/email), загрузкой файлов и AI-агентом для автоматического разбора переписки.

**Сверка 2 — Финальная по инвойсу («Сверка заказа»)**  
Поставщик EU присылает финальный инвойс + упаковочный лист (packing list). Наша команда выезжает на склад EU и руками сверяет физический товар с упаковочным листом прямо на месте. При расхождениях — запрашиваем коррекцию инвойса. Реализована как модал с загрузкой документов и AI-распознаванием.

**Контрольный лист — физическая приёмка на складе EU**  
Отдельный модуль (`page-checklists`). Во время физической сверки на складе EU открываем КЛ и отмечаем фактически присутствующие и отсутствующие позиции. Когда КЛ переходит в статус «Завершён» — система разблокирует кнопку «В завершённые» в карточке заказа.

### Перевод в «Завершённые»

**Условие:** Контрольный лист, связанный с заказом, должен иметь статус **«Завершён»**.  
**Кто переводит:** Оператор вручную (кнопка «В завершённые» в шапке заказа).  
**Автоматически не переносится.**  
После перевода заказ исчезает из секции «Размещённые» и появляется в секции «Завершённые» с датой и процентом выполнения.

### Статусы позиций в заказе

| Статус | Цвет | Источник |
|---|---|---|
| Размещено | синий | ставится при создании заказа |
| Подтверждено | серый | поставщик подтвердил позицию |
| Прибыло | зелёный | позиция физически получена |
| Отгружено | фиолетовый | отгружено, трекинг не подтверждён |
| Перенесена | оранжевый | перенесена в следующую поставку |
| Отменен | красный | позиция отменена поставщиком |

> **Важно:** статусы «Отгружено», «Перенесена», «Отменен» подгружаются с Контрольного листа — вручную не выставляются.

### `page-orders` — список заказов

Три секции:
- **Активные** — заказы на актуализации/согласовании
- **Размещённые** — заказы, по которым поставщик прислал проформу. Содержит виджет бэкордера.
- **Завершённые** — заказы с завершённым КЛ, переведённые оператором. Показывает % выполнения снаружи.

**Типы заказов (чипы):**
- `STD` — стандартный заказ (синий)
- `SO` — стандартный заказ John Deere (бирюзовый)
- `AN` — стандартный John Deere 4 недели (фиолетовый)
- `BCK` — Backorder / перезаказ (алый, `#C0192B`)
- `MD` — Express Machine Down (индиго)
- `MD USA` — Express John Deere из США (индиго)
- `EXPR` — Express Manitou/AGCO (пурпурный)
- `CNCL` — отменён (красный)

## 3. Backorder

**Бизнес-логика:**  
Бэкордер — позиции из заказов, которые поставщик не смог поставить в срок. Процесс работы с бэкордером:

1. Закупщик (раздел Поставщик) видит причину отмены/задержки по каждой позиции
2. Закупщик сообщает менеджеру в рамках заказа и описывает возможные решения: заказать с другого склада, экспресс-поставка, перезаказ под другим кодом и т.д.
3. Менеджер получает извещение внутри своего заказа, согласовывает с клиентом изменившиеся обстоятельства
4. Менеджер возвращает ответ по данной позиции обратно внутри заказа (согласован ли вариант решения)
5. Закупщик (раздел Поставщик) на основе ответа может перезаказать позиции — создаётся новый заказ типа BCK, объединяющий позиции по поставщику

**Статус менеджера (mgrStatus) — колонка «Менеджер»:**

| Значение | Бейдж | Цвет | Смысл |
|---|---|---|---|
| `agree` | Согласование | оранжевый | ожидает решения |
| `update` | Обновление | синий | требует обновления ETA |
| `ordered` | В заказе | зелёный | перезаказано у поставщика |

Бейдж BCK-XXXX (алый `#C0192B`) — появляется в строке когда есть перезаказ.

**Секция «Перезаказано»** — нижняя часть таблицы, отделена визуально. Показывает позиции со статусом `ordered` + номер BCK.

### BCK-цикл (кнопка «Перезаказать»)

1. Закупщик нажимает **Перезаказать** → `boReorderOpen()` → modal-reorder
2. В модале: список всех BO с чекбоксами, бренд, поставщик
3. Выбирает позиции → `boReorderUpdatePreview()` группирует по поставщику → превью BCK-номеров
4. Нажимает ОК → `boReorderConfirm()`:
   - Для каждой группы поставщика создаётся BCK-XXXX (счётчик `bckSeq`)
   - Обновляется `BO_DATA[art].bckOrder` и `mgrStatus = 'ordered'`
   - Позиции появляются в секции «Перезаказано»
   - BCK-заказы появляются в разделе Заказы → Размещённые с типом-чипом `BCK`

### `page-bo-detail` — детальная страница позиции бэкордера

## Бизнес-логика: многозаказный инвойс

Один инвойс поставщика может покрывать несколько проформ (например, 5 шт из PRF-089 и 2 шт из PRF-088 → итоговый инвойс на 7 шт по одному артикулу).

**Алгоритм сопоставления:**

1. Берём артикул + итоговое количество из инвойса
2. Ищем совпадение по артикулу в активных заказах этого поставщика
3. Если суммарное количество по заказам равно количеству в инвойсе — распределяем автоматически
4. Если один артикул с одинаковым количеством есть в двух заказах — ставим флаг для ручного разбора

**Пример дубля RE522878:**
- PRF-2026-089: заказано 295 шт
- PRF-2026-088: заказано 210 шт
- В инвойсе: 505 шт по RE522878
- Система: 505 = 295 + 210 → автоматически разбивает по заказным количествам

**Отображение в UI:**
- `modal-reconcile` STATE 3: жёлтый баннер предупреждения при обнаружении многозаказного инвойса
- `reconcileConfirm()`: расширенный тост (6 сек) с деталями распределения + список затронутых заказов
- `page-completed-order`: инвойс в разделе «Документы» с бейджем «N заказов» (оранжевый)
- `coToggleCycle()`: в панели «Полный цикл» объясняется логика разбивки дублей
